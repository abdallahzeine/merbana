# Alembic Workflow

## Policy
- Alembic revisions are the only approved path for schema changes.
- Manual schema edits in SQLite are not allowed.
- Autogenerated migrations must be reviewed and corrected before merge.

## Source Of Truth For Database URL
The migration and runtime paths share one DB URL source:
1. `MERBANA_DB_URL` (if set)
2. Fallback to backend path resolver (`backend.paths.get_database_url`)

Examples:
- Linux: `sqlite:////home/user/Desktop/POS/data/merbana.db`
- Windows: `sqlite:///C:/Users/name/Desktop/POS/data/merbana.db`

## Alembic Location
- Config: `Deployment/backend/alembic.ini`
- Environment: `Deployment/backend/alembic/env.py`
- Revisions: `Deployment/backend/alembic/versions/`

## Command Reference

### Linux / Bash
Set DB URL and upgrade:
```bash
export MERBANA_DB_URL='sqlite:////home/user/Desktop/POS/data/merbana.db'
python3 -m alembic -c Deployment/backend/alembic.ini upgrade head
```

Stamp migrated DB to baseline (Task 5 handoff):
```bash
export MERBANA_DB_URL='sqlite:////home/user/Desktop/POS/data/merbana.db'
python3 -m alembic -c Deployment/backend/alembic.ini stamp 0001_initial_schema
```

### Windows / PowerShell
Set DB URL and upgrade:
```powershell
$env:MERBANA_DB_URL = 'sqlite:///C:/Users/name/Desktop/POS/data/merbana.db'
python -m alembic -c Deployment/backend/alembic.ini upgrade head
```

Stamp migrated DB to baseline (Task 5 handoff):
```powershell
$env:MERBANA_DB_URL = 'sqlite:///C:/Users/name/Desktop/POS/data/merbana.db'
python -m alembic -c Deployment/backend/alembic.ini stamp 0001_initial_schema
```

## Backup Gate (Mandatory Before Upgrade)
Create a timestamped backup before every `upgrade`:
- `merbana.db`
- `merbana.db-wal` (if present)
- `merbana.db-shm` (if present)

Suggested timestamp format: `YYYYMMDD_HHMMSS`.

### Linux backup example
```bash
TS="$(date +%Y%m%d_%H%M%S)"
mkdir -p "$HOME/Desktop/POS/backups/$TS"
cp "$HOME/Desktop/POS/data/merbana.db" "$HOME/Desktop/POS/backups/$TS/"
cp "$HOME/Desktop/POS/data/merbana.db-wal" "$HOME/Desktop/POS/backups/$TS/" 2>/dev/null || true
cp "$HOME/Desktop/POS/data/merbana.db-shm" "$HOME/Desktop/POS/backups/$TS/" 2>/dev/null || true
```

### Windows backup example
```powershell
$ts = Get-Date -Format "yyyyMMdd_HHmmss"
$backup = "$env:USERPROFILE/Desktop/POS/backups/$ts"
New-Item -ItemType Directory -Force -Path $backup | Out-Null
Copy-Item "$env:USERPROFILE/Desktop/POS/data/merbana.db" -Destination $backup -Force
if (Test-Path "$env:USERPROFILE/Desktop/POS/data/merbana.db-wal") { Copy-Item "$env:USERPROFILE/Desktop/POS/data/merbana.db-wal" -Destination $backup -Force }
if (Test-Path "$env:USERPROFILE/Desktop/POS/data/merbana.db-shm") { Copy-Item "$env:USERPROFILE/Desktop/POS/data/merbana.db-shm" -Destination $backup -Force }
```

## Restore Instructions
1. Stop app/launcher.
2. Copy backup DB files back to the active data directory.
3. Start the app again.

## Task 5 -> Task 9 Integration
When a database was produced by Task 5 JSON migration:
1. Run migration script and validate generated artifacts.
2. Do not rebuild data through Alembic.
3. Stamp database to baseline revision:
   - `stamp 0001_initial_schema`
4. Use `upgrade head` for all later schema revisions.

## Clean Install Path
For a fresh empty DB:
1. Set `MERBANA_DB_URL`.
2. Run `upgrade head`.
3. Start backend/app.

## Packaged Path Notes
In packaged deployments, database files typically live under a data directory beside the executable. Always point `MERBANA_DB_URL` to that packaged data path, not to a repository path.

## Deployment/Startup Strategy
Automatic startup migration is not enabled in runtime code.
Operator step is required before app use when schema has changed.
