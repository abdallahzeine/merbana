# Task 9 - Set Up Alembic Migrations

## Objective
Introduce a controlled, reviewable schema-evolution workflow for the SQLite backend.

## Prerequisite
- [ ] Tasks 2 and 3 are complete and models are stable enough for an initial baseline revision.

## Codebase Anchors
- `requirements.txt`
- backend package (for example `Deployment/backend/`)
- `build_windows.py`
- `Deployment/build_linux.py`

## Migration Workflow Rules
- [x] Alembic revisions are the only allowed path for future schema changes; manual SQLite schema edits are not allowed.
- [x] Autogenerated migrations must be reviewed and corrected manually before acceptance.
- [x] Runtime database path and CLI migration path must resolve through one documented source of truth.

## Checklist
- [x] Add Alembic dependency and pin version in `requirements.txt`.
- [x] Initialize Alembic inside the backend package directory.
- [x] Configure `alembic.ini` and `env.py` to import SQLAlchemy metadata.
- [x] Resolve runtime-vs-CLI DB path conflict by using an environment-variable DB URL for Alembic commands (for example `MERBANA_DB_URL`).
- [x] Create the initial revision from finalized models and run upgrade on a clean DB.
- [x] Define the explicit connection between Task 5 migrated data and Alembic: after migration, stamp the database to the initial head revision instead of rebuilding data.
- [x] Add migration command docs for Windows and Linux flows, including shell syntax differences and packaged-path considerations.
- [x] Ensure build/deploy/startup flow applies pending migrations before app use begins, or document an explicit operator step if auto-upgrade is rejected.
- [x] Define backup gate: copy the SQLite DB file before every upgrade, with timestamped naming and restore instructions.
- [x] Define migration review checklist including schema diff review, downgrade strategy, and expected data-impact assessment.

## Acceptance Criteria
- [x] A fresh install can reach head revision from zero.
- [x] A migrated database can be stamped/upgraded without data loss.
- [x] Windows and Linux deployment flows document the same logical migration process.

## Deliverable
- [x] Versioned Alembic workflow with baseline revision, documented commands, backup gate, and deployment integration strategy.

## Library Choices with Justification
- `Alembic` for schema-version control because it is the migration tool designed to track `SQLAlchemy` metadata and is safer than maintaining hand-written upgrade scripts.
- `SQLAlchemy 2.x` metadata as the migration source because keeping model definitions and revision generation aligned is less error-prone than duplicating schema intent in separate DDL files.
- `python-dotenv` is explicitly rejected for this task because deployment already has clear environment-variable entry points and adding another configuration layer would obscure the DB URL source of truth.

## Architecture Decisions
- Chosen: Alembic revisions as the only schema-change path. Rejected: manual edits to the SQLite file. Why: reproducible upgrades and rollbacks require a reviewed migration history.
- Chosen: environment-variable DB URL resolution for CLI migration commands. Rejected: hard-coding paths separately in runtime and Alembic config. Why: two path-resolution systems would eventually diverge across Windows and Linux packaging.
- Chosen: stamp migrated databases to the baseline revision after Task 5. Rejected: rebuilding migrated production data from scratch under Alembic just to create a version table. Why: version control should attach to validated migrated data, not force a second risky import.
